How We'd Validate the Redesign — Two Lightweight Studies

Author: Andres G., UX Designer For discussion with: Research & Manager Prepared: June 16, 2026 Companion to: Goal & Validation Plan (doc 28)
Purpose of this doc

What we'd run, in plain terms — so we can right-size it together

Two complementary studies de-risk the redesign: one checks whether the structure (the taxonomy) makes sense, the other checks whether the panel works in the hand. Each answers a different question, and neither is a heavy research program.

Right-sized, not a deep research stage. Study 1 is a remote, unmoderated test that needs no prototype and no scheduling — results come back in days. Study 2 is exactly the "put a user on the prototype and watch" session the team is comfortable with. If we only have appetite for one touch, Study 2 is the minimum — but Study 1 is cheaper and de-risks the most expensive decision (taxonomy), so doing both is the small-effort, high-coverage option.

Study 1 · Validates structure

Tree testing

Remote · Unmoderated · No prototype

Can people find things in the proposed structure?

A "reverse card sort" — we give users the category tree as plain text and ask them to find where they'd go for a task.
What it is

Users see only the navigation hierarchy as a clickable text outline (no colors, no layout, no panel). For each task they drill down and pick where they'd expect to find the item. Because there's no visual design to react to, it isolates one thing: does the structure make sense?

What it helps prove

Whether Level 2 (grouped) genuinely helps people find products versus today's structure — and whether grouping is worth the catalog team's content work. It directly answers the taxonomy decision, which is the most expensive and least reversible part of the project.

How many users & which categories

~30–50 participants per variant. We test three trees on the same tasks: current · Level 1 (reorder + add) · Level 2 (grouped). The current tree is our built-in baseline — the comparison analytics never gave us. The full department set is in each tree; the ~8–12 tasks sample high-traffic menus, the HVAC vocabulary gap, and a cross-department duplicate.

How it's conducted

Built in a tree-testing tool (Treejack / Optimal Workshop or Lyssna), sent as a link, completed unmoderated over a few days. No moderator time, no scheduling. We segment respondents (trade pro vs. homeowner) with one screener question so we can read results by audience.

Example tasks
"You need a circulator pump for a residential boiler. Where would you click to find it?"Tests the Heating grouping + the "Circulators & Pumps" relabel
"You're looking for a recovery machine to pull refrigerant. Where would you go?"Tests the HVAC/Tools vocabulary gap + cross-link
"Find PVC cement and primer."Tests the Plumbing label fix (from the old "Chemicals")
Study 2 · Validates interaction

Moderated usability test on the prototype

Moderated · Think-aloud · On the prototype

Does the menu actually work when someone uses it?

The "put a user on the prototype and watch" session — exactly the hands-on test the team wants.
What it is

A facilitator gives a participant realistic tasks on the clickable prototype and asks them to think aloud. We watch where they click, where they hesitate, and where they get stuck — then ask a quick ease rating after each task.

What it helps prove

That the interaction model holds up: click-to-open (the Baymard #1 fix), the column reveal, whether users can tell where they are now that the breadcrumb is gone (the Baymard #3 / scope-cue question), and whether "Shop by Job" gets used or ignored. Tree testing can't see any of this — it needs eyes on behavior.

How many users & which categories

5–6 per segment — trade pros and homeowners run separately, because their mental models differ — so ~10–12 total. (Five users per group surfaces the large majority of usability issues.) We don't test all eight departments; we pick a representative subset: one large grouped menu (HVAC or Fittings), one small one (Valves), the Shop All open/close flow, and Shop by Job.

How it's conducted

30–45 min sessions, in person or via screen-share. Each session opens with a quick first-click task (first click strongly predicts overall success), then moves to think-aloud tasks, then the ease questions. A researcher moderates; the designer observes and takes notes. We run one pilot session first to shake out the script.

Example tasks
"Starting from the homepage, find where you'd shop for a condensate pump, then show me how you'd narrow to 230V models."Tests Shop All open → HVAC group → Column 3 filters
"You want everything you'd need for a boiler replacement. Show me how you'd approach that."Tests whether "Shop by Job" is discovered and used
"You realize you opened the wrong department. Get yourself to Valves instead."Tests scope clarity + switching (the removed-breadcrumb question)
Scope guardrails

What we're deliberately not doing

Out of scope (keeps it light)

  • No large-scale survey or diary study
  • No live A/B test now — that's a post-launch monitor once tracking ships, not a gate
  • No full-catalog walkthrough — we sample categories, not all eight in depth
  • No eye-tracking or lab equipment

Why this is enough

  • Tree test gives quantitative confidence on the costly taxonomy call — cheaply
  • Usability catches the interaction problems numbers can't explain
  • Together they cover both risk axes (structure + interaction) in ~2–3 weeks
  • Both are re-runnable post-launch as a benchmark once the menu is instrumented
How we'll read the results

What "good" looks like

MeasureFrom which studyWhat we're looking for
Task success rateTree test + usabilityHigher than today's structure (tested in the same study). A common target is ~80%+ on core findability tasks.
DirectnessTree testUsers go straight to the right place rather than backtracking — fewer wrong turns than the current tree.
Time-to-findTree testFaster than the current structure on the same tasks.
Single-Ease Question (SEQ)UsabilityPer-task ease rating trending high (e.g., ≥5.5 of 7); flags tasks that "worked" but felt hard.
Observed frictionUsabilityWhere people hesitate, misread a label, or lose their place — the "why" behind the scores.

The headline we'll be able to report is comparative — e.g. "task success rose from N% on today's structure to N% on Level 2, with fewer wrong turns" — which is what makes a result persuasive even though the live menu was never tracked.

Logistics

What we'd need, and a rough order

To run these we need

  • The category trees as text (current + Level 1 + Level 2) — already drafted in doc 18
  • A stable clickable prototype for the subset of menus we test
  • ~10–12 participants for Study 2, screened into the two segments
  • A tree-testing tool license (Optimal Workshop / Lyssna) for Study 1
  • A short task script + ease questions (designer drafts, researcher refines)

Suggested order (~2–3 weeks)

  • Week 1: Tree test live (runs in the background, unmoderated)
  • Week 1–2: Recruit + pilot the usability session
  • Week 2–3: Run usability sessions; read tree results
  • End: One combined readout — structure verdict + interaction fixes
If we can only do one: start with the tree test. It's the cheapest to run, needs no prototype, and de-risks the decision that's most expensive to get wrong (the taxonomy / catalog content roadmap). Add the usability sessions when the prototype is stable.
To decide with Research & Manager

Open questions for this conversation

Scope
Both studies, or start with the tree test alone and add usability once the prototype is ready?
Recruiting
Can we source trade pros and homeowners from existing customer lists / a panel, and is incentive budget available for the moderated sessions?
Coverage
Which large menu anchors Study 2 — HVAC (vocabulary gap) or Fittings (biggest, most complex)? And do we test the Shop All + Shop by Job header concept here too?
Tooling
Do we already have a tree-testing tool, or should we budget for one? Who owns moderating versus observing?
Thresholds
Agree the success bar up front (e.g., ~80% task success, SEQ ≥5.5) so the readout has a clear pass/fail rather than a debate after the fact.
